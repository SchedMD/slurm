Running Slurm with TPUs

To run Slurm workloads with TPU (Tensor Processing Unit) accelerators on Kubernetes via the Slinky Slurm operator, you must first install Slinky on your cluster. For detailed setup steps, see the Slinky Installation Guide.

Worker NodeSet Configuration

To configure your worker nodes to request TPUs, update the worker NodeSet and Controller in your Helm values file (e.g., values.yaml) to request TPU resources and specify the appropriate node selectors and tolerations: 

nodesets:
  tpu:
    enabled: true
    scalingMode: StatefulSet
    replicas: 1
    slurmd:
      image:
        repository: gcr.io/gke-release/slinky/slurmd
        tag: 25.11-ubuntu24.04-gke.9
      resources:
        limits:
          google.com/tpu: 8
        requests:
          google.com/tpu: 8
    podSpec:
      nodeSelector:
        cloud.google.com/gke-tpu-accelerator: tpu-v5-lite-podslice
        cloud.google.com/gke-tpu-topology: 2x4
      tolerations:
        - key: google.com/tpu
          operator: Exists
          effect: NoSchedule
controller:
  slurmctld:
    image:
      repository: gcr.io/gke-release/slinky/slurmctld
      tag: 25.11-ubuntu24.04-gke.9
    resources:
      requests:
        cpu: 500m
        memory: 512Mi
  # Configure the TPU partition in slurm.conf
  extraConf: |
    PartitionName=tpu Nodes=tpu State=UP SuspendTime=INFINITE Oversubscribe=Exclusive PowerDownOnIdle=NO ResumeTimeout=300 SuspendTimeout=120

TPU Environment Variable Propagation

A key difference when running TPU workloads under Slurm compared to direct Kubernetes execution is environment variable propagation. Kubernetes automatically sets all necessary TPU-related environment variables (e.g., TPU_ACCELERATOR_TYPE, TPU_TOPOLOGY) in the worker pods. However, Slurm jobs executed via sbatch do not automatically inherit these pod-level environment variables. Without them, JAX or other TPU-enabled frameworks will fail to initialize or time out while attempting to locate the accelerator.

To address this, you must explicitly capture the TPU environment variables from the compute nodes (e.g., via env | grep TPU) and export them inside your Slurm job submission script.

Below is an example of a Slurm batch wrapper script (runner.sh) for a single-host TPU setup (such as a TPU v5e with 8 chips):

#!/bin/bash
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -p tpu
#SBATCH --job-name=tpu-test
#SBATCH --output=tpu-test-%j.out

# TPU environment variables (captured from the compute node environment)
export TPU_ACCELERATOR_TYPE=v5litepod-8
export TPU_TOPOLOGY=2x4
export TPU_SKIP_MDS_QUERY=true
export TPU_CHIPS_PER_HOST_BOUNDS=2,4,1
export TPU_HOST_BOUNDS=1,1,1
export TPU_TOPOLOGY_WRAP=false,false,false
export TPU_TOPOLOGY_ALT=false
export TPU_RUNTIME_METRICS_PORTS=8431,8432,8433,8434,8435,8436,8437,8438

# JAX single-host configurations
export JAX_COORDINATOR_ADDRESS="127.0.0.1:8470"
export TPU_WORKER_ID=0
export TPU_WORKER_HOSTNAMES="localhost"

ulimit -n 65536

# Execute the user command
srun python3 $@

Example JAX Job

Below is an example JAX Python script (simple_tpu.py) that initializes the distributed runtime and prints the detected TPU devices:

#!/usr/bin/env python3
import jax

# Initialize the JAX distributed system
jax.distributed.initialize()

print("--- JAX Distributed Initialized ---")
print(f"Process Index (Node ID): {jax.process_index()}")
print(f"Total JAX Processes: {jax.process_count()}")
print(f"Local Device Count (per host): {jax.local_device_count()}")
print(f"Total Device Count (global): {jax.device_count()}")
print(f"Available Devices: {jax.devices()}")
print("----------------------------------")

Ensure that JAX (with TPU support) is installed in your worker node environment, then submit the job from the login node using:

kubectl exec -n slurm slurm-cluster-login-slinky-xxx -c login -- bash -c "cd /tmp && sbatch runner.sh /tmp/simple_tpu.py"
